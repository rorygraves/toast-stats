How the sprint runner works

One launchd tick, end to end. Cron fires every :17/:47; the script resolves which work item to execute, spawns an isolated Claude session in its own git worktree, watches the work ship, and either cascades to the next epic or waits for the next tick. Every numbered stage below was reshaped by at least one bug discovered in live operation.

Click any boxed node to see what it does, which bugs were fixed there, and which lessons document the design rationale.

1

Schedule

every :17, :47

launchd fires the script on a calendar interval. The plist captures env vars (META_EPIC, STRICT_GATE) so the script knows its target.

↓
2

Reconcile

first action of every tick

Acquire a lock, sweep orphan worktrees, rotate the log. The reconcile step is convergent — any leak from prior ticks gets cleaned up here.

↓
3

Resolve

META_EPIC → Epic → Sprint

Three-level hierarchy. Each level is just a GitHub issue with a markdown checklist. The script walks them top-down to find the first unchecked sprint.

↓
4

Gate

predecessor verification

Sprint N can only launch if Sprint N-1's issue is CLOSED. Under STRICT_GATE=1, also requires sprint-verified label. Cold-start sprints (Sprint 1) bypass.

↓
5

Spawn

isolated session per sprint

Create a dedicated git worktree at ~/sprint-worktrees/sprint-<issue>. Spawn a detached screen running claude --remote-control sprint-<issue> inside it. Verify launch via pgrep.

↓
6

Work

inside the spawned session

The Claude session reads its bootstrap, re-confirms predecessor + prod reachability, then invokes /sprint to drive Red → Green → Refactor against the work item.

↓
7

Verify

live, on production

After PR merge: drive the actual change on ts.taverns.red via a temporary Playwright smoke. Capture screenshots. Corroborate with a data-path check on the prod CDN.

↓
8

Closeout

strict order matters

Apply the sprint-verified label → tick the Sprint N checkbox → close the sub-issue → if this was the last sprint, close the epic too with a summary comment. Order matters: ticking-before-closing eliminates the duplicate-launch race.

↓
9

Reap

next tick detects completion

On the next tick, the runner sees a screen whose paired sub-issue is CLOSED + checkbox ticked. Reaps the screen, force-kills orphan claude pids, removes the worktree, prunes the auto-branch if merged.

↓
10

Cascade

no idle ticks

If the active epic has no unchecked sprints remaining, advance_meta_epic ticks its line in META_EPIC. The script doesn't exit — it loops back to Resolve and tries to launch the next epic's first sprint in the same tick.

↻ loops back to Stage 3 with a freshly-resolved next epic
